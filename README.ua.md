# Dental Care App

**Мова:** [English](README.md) | Українська

## 1. Огляд проєкту

Dental Care App - це мобільний застосунок на **React Native CLI**, створений як MVP для стоматологічної клініки. Він допомагає пацієнтам проходити основний шлях догляду в одному місці: onboarding, авторизацію, вибір послуги, запис до лікаря, історію візитів і контроль прийому ліків після лікування.

Мета проєкту - зробити стоматологічний догляд простішим для пацієнтів і більш структурованим для клініки. Користувачі можуть переглядати лікарів, обирати стоматологічні послуги, бронювати прийоми, переглядати попередні візити та дотримуватися призначеного плану лікування.

У майбутньому продукт можна розширити CRM web-версією для персоналу клініки, де адміністратори й лікарі зможуть керувати графіками, записами, пацієнтами, планами лікування та статистикою роботи клініки.

## 2. Аналіз наявного застосунку

Застосунок уже має повноцінний мобільний сценарій із авторизацією, навігацією, головним dashboard, записом на прийом, профілем користувача, історією візитів і секцією контролю прийому ліків.

### Ключові функції

- **Onboarding** - вступні екрани, які пояснюють призначення застосунку.
- **Авторизація та реєстрація** - вхід, реєстрація та підтвердження OTP.
- **Головний екран** - статистика візитів, найближчий запис, швидкі дії, банери та блок довіри.
- **Запис на прийом** - вибір послуги, лікаря, дати, часу та підтвердження бронювання.
- **Профіль лікаря** - інформація про лікаря, рейтинг, освіта, відгуки та дія для запису.
- **Історія візитів** - минулі й майбутні записи з переходом до деталей візиту.
- **Medications** - огляд плану лікування, щоденний графік прийому ліків і відстеження прогресу.
- **Профіль користувача** - персональна інформація та основні дії акаунта.

### Основні сценарії використання

1. Користувач відкриває застосунок, проходить onboarding і входить в акаунт.
2. Користувач переглядає на головному екрані статистику візитів, майбутні записи та швидкі дії.
3. Користувач запускає сценарій бронювання, обирає послугу, лікаря, дату та час.
4. Після підтвердження запис зберігається й відображається в історії візитів.
5. Якщо існує план лікування, користувач може відстежувати ліки та відмічати прийняті дози.

### Зони для розширення функціональності

- **Покращення навігації** - швидкі дії мають вести прямо до ключових сценаріїв: бронювання, підтримки в чаті та історії візитів.
- **Секція підтримки та FAQ** - вкладку чату можна розширити до практичного розділу з відповідями на типові питання пацієнтів.
- **CRM для клініки** - майбутній web-інтерфейс для керування лікарями, графіками, пацієнтами, записами та аналітикою.

## 3. Розширення функціоналу

### Додана функціональність: Medications

Секцію **Medications** було додано, щоб допомогти пацієнтам дотримуватися плану лікування після стоматологічної процедури.

Функція включає:

- період лікування;
- поточний день лікування;
- загальний прогрес лікування;
- список ліків, запланованих на сьогодні;
- дозування та час прийому;
- можливість відмітити препарат як прийнятий;
- перегляд повного плану лікування.

Це покращує користувацький досвід, тому що застосунок підтримує не лише запис до лікаря, а й процес відновлення після візиту.

### Навігація Quick Actions

Головний екран містить картки швидких дій для основних задач користувача:

- **Book Appointment** - вхід до сценарію бронювання;
- **Find Doctor** - вхід до вибору лікаря;
- **Chat Support** - вхід до допомоги та підтримки;
- **View Records** - вхід до історії візитів.

Ці дії роблять головний dashboard практичнішим і зменшують кількість кроків до важливих розділів.

## 4. Керування станом

У застосунку використовуються різні підходи до керування станом залежно від типу даних.

### Context API

**Context API** використовується для глобального стану застосунку:

- `AuthContext` - стан авторизації, профіль користувача, login/logout і OTP-реєстрація;
- `ThemeContext` - стан поточної теми та перемикання теми.

Context API добре підходить для цього, тому що ці значення використовуються в різних частинах застосунку, але не потребують складної Redux-архітектури.

### React Query

**TanStack React Query** використовується для серверних даних:

- лікарі;
- записи;
- доступні слоти;
- історія візитів;
- плани лікування;
- прийом ліків.

React Query добре підходить для цього проєкту, тому що працює з кешуванням, станами завантаження, помилками та оновленням серверних даних без зайвої складності глобального client-state.

## 5. Компонентний підхід

Проєкт побудований за компонентним принципом. Повторно використовувані елементи інтерфейсу винесені в окремі файли або папки, що спрощує підтримку й розширення коду.

### Приклади компонентів

- `AppointmentCard` - відображає найближчий запис.
- `EmptyAppointmentCard` - показує empty state, коли немає майбутніх записів.
- `MedicationReminder` - показує короткий підсумок прийому ліків на головному екрані.
- `MedicationItem` - відображає елемент графіка прийому ліків.
- `DoctorCard` - показує інформацію про лікаря в сценарії бронювання.
- `ServiceCard` - показує варіант стоматологічної послуги.
- `QuickActionCard` - відображає швидку дію на головному екрані.
- `CustomBtn` - повторно використовувана кнопка з різними візуальними станами.
- `TrustBlock` - показує інформацію про довіру та безпеку.

## 6. UX/UI вдосконалення

Застосунок покращує користувацький досвід завдяки:

- зрозумілій нижній tab-навігації;
- плавній анімації header на головному екрані;
- повторно використовуваним карткам для записів, лікарів, послуг і ліків;
- єдиній палітрі кольорів, відступам, тіням і заокругленим елементам;
- станам loading, empty та content для екранів, які залежать від даних;
- типізованій передачі параметрів між екранами, наприклад:
  - `DoctorList` -> `DoctorProfile`;
  - `DoctorProfile` -> `SelectDate`;
  - `SelectDate` -> `BookingConfirm`;
  - `VisitHistory` -> `VisitDetails`.

## 7. Структура проєкту

```txt
src/
  api/                 # Firebase API та запити до даних
  assets/              # іконки, зображення, скріншоти
  components/          # великі повторно використовувані компоненти
  constants/           # маршрути, тема, колекції, статичні дані
  context/             # AuthContext, ThemeContext
  hook/                # кастомні hooks
  interfaces/          # TypeScript типи
  layout/              # спільні layout-компоненти
  mockData/            # локальні тестові дані
  navigation/          # Root, Tab та Stack навігація
  screens/             # екрани застосунку
  ui/                  # маленькі UI-компоненти
  utils/               # допоміжні функції
```

## 9. Огляд презентації

- **Слайд 1: Основна ідея** - Dental Care App організовує шлях пацієнта від вибору лікаря до контролю лікування після візиту.
- **Слайд 2: Ключові функції** - авторизація, перегляд лікарів, запис на прийом, вибір дати й часу, історія візитів, профіль користувача та нагадування про ліки.

- **Слайд 4: Структура та зручність** - модульна структура розділяє екрани, компоненти, hooks, API-логіку, контексти й утиліти, що полегшує підтримку та розширення проєкту.

## 10. Майбутній функціонал

Застосунок можна розширити функціями, які покращать досвід як для пацієнтів, так і для стоматологічної клініки:

- **Push-нотифікації** - нагадування про прийом ліків і майбутні візити до лікаря.
- **FAQ / Chat Support** - розділ допомоги з відповідями на типові питання пацієнтів до та після візиту.
- **AI-рекомендації часу** - рекомендовані слоти запису на основі завантаженості лікаря, тривалості процедури та контексту пацієнта.
- **CRM web-версія** - dashboard для клініки, щоб керувати пацієнтами, лікарями, графіками, записами та статистикою.
- **Аналітика лікування** - відстеження того, наскільки стабільно пацієнти дотримуються планів прийому ліків і післяпроцедурних рекомендацій.

Проєкт краще відображає реальний сценарій стоматологічного застосунку для пацієнтів і має чіткий напрям для подальшого розвитку.
